How these interviews go

Reported GPU-role coding rounds (NVIDIA, AMD, AI labs) sit around LeetCode-medium difficulty but in CUDA: write a working kernel on a whiteboard or shared editor, then defend its performance — coalescing, shared-memory use, synchronization, and how you'd verify and profile it. The recurring set is remarkably stable: reduction, transpose/rotation, tiled matmul, histogram, scan, stencil, softmax, a debugging exercise, and host-side transfer pipelining. The production set adds stable compaction, sparse/ragged work, fused normalization, and dynamic quantization. Each problem below states the task, what the interviewer is actually testing, and a complete solution with the discussion points that separate a pass from a strong pass. Write your attempt before opening the solution.

Interview contract: state launch and input assumptions, write a CPU reference, test empty/tiny/tail shapes plus adversarial values, run Compute Sanitizer, and benchmark only after correctness passes. Empty work is a host-side no-op (or explicit identity); never issue a zero-grid launch.

Warm-ups

Five-minute questions that screen for the basics: index math, guards, launch configuration.

1. SAXPY (vector add) — the screener

Warm-up

Tests: thread indexing, bounds guard, grid-stride idiom, launch configuration.

Write a kernel computing y[i] = a·x[i] + y[i] for n floats, correct for any n, plus the host-side launch. Then: how do you pick the grid size?

Acceptance tests: n ∈ {0,1,31,32,255,256,257,1M}; mixed-sign and large/small values; CPU-FMA comparison with tolerance; allocation canaries remain unchanged.

Show solution & discussion
__global__ void saxpy(int n, float a, const float* __restrict__ x,
                      float* __restrict__ y) {
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n;
         i += gridDim.x * blockDim.x)              // grid-stride loop
        y[i] = a * x[i] + y[i];
}

// host
int block = 256;
if (n > 0) {                                       // a zero-block launch is invalid
    int grid = std::min((n + block - 1) / block,   // round UP, never n/block
                        32 * numSMs);               // cap: a few waves is enough
    saxpy<<<grid, block>>>(n, a, d_x, d_y);
}
  • The classic fails: n / block drops the tail, no guard writes out of bounds, and launching a zero-sized grid for n=0 is invalid. The host treats empty input as a no-op; the grid-stride loop handles every nonempty tail.
  • Grid size answer: enough blocks to fill every SM a few times over (e.g. 32 × SM count); more than that adds nothing for a memory-bound kernel. Mentioning the cap shows you know launches aren't "one thread per element or bust".
  • Performance framing: SAXPY is 2 FLOPs per 12 bytes — hopelessly memory-bound, so the only meaningful metric is achieved bandwidth vs peak. Saying this unprompted is a strong-pass signal.

2. AoS → SoA conversion

Warm-up

Tests: whether you actually understand coalescing, not just recite it.

Given struct alignas(16) P { float x, y, z, w; } stored as an array of n structs, write a kernel that produces four separate arrays X, Y, Z, W. Which direction of this kernel is coalesced, and does it matter here?

Acceptance tests: n ∈ {0,1,31,32,33} with distinct bit patterns per field; exact output comparison; require sizeof(P)==16 and 16-byte source alignment, then inspect generated code for the intended vector load.

Show solution & discussion
struct alignas(16) P { float x, y, z, w; };
static_assert(sizeof(P) == 16);

__global__ void aos_to_soa(const P* __restrict__ in,
                           float* __restrict__ X, float* __restrict__ Y,
                           float* __restrict__ Z, float* __restrict__ W,
                           int n) {
    for (int i = blockIdx.x * blockDim.x + threadIdx.x; i < n;
         i += gridDim.x * blockDim.x) {
        P p = in[i];             // eligible for one aligned 16-byte load;
                                 // confirm LDG.128 (or equivalent) in SASS
        X[i] = p.x;  Y[i] = p.y; // four coalesced 4-byte stores
        Z[i] = p.z;  W[i] = p.w;
    }
}
  • The insight being probed: an aligned whole-record load lets a warp request 32 consecutive 16-byte records = 512 contiguous bytes. Loading field-by-field can instead issue four stride-16 instructions with poor sector utilization, so inspect SASS rather than assuming the compiler vectorized it.
  • Requirement to state: the stored type itself is the 16-byte-aligned P. Alignment alone does not make reinterpreting an unrelated P* as float4* legal under C++ aliasing rules.
  • Bigger picture: this kernel is worth running once if many later kernels each use one field; if only one kernel reads the data, fuse the conversion into it instead — a data-layout decision, not a kernel decision.

3. Parallel sum reduction

Most asked

Tests: tree reduction, __syncthreads discipline, warp shuffles, cross-block combination.

Sum n floats into a single value. Start with a working version, then the interviewer will push: fewer barriers? less shared memory? how do blocks combine?

Acceptance tests: launch exactly 256 threads; host handles n=0 as the additive identity; test {1,31,32,255,256,257,1M}, zeros and alternating magnitudes; initialize the output; compare with abs/rel tolerance across repeated runs.

Show solution & discussion
__global__ void reduce_sum(const float* __restrict__ in, float* out, int n) {
    __shared__ float s[256];                       // = blockDim.x
    // 1) each thread accumulates many elements in a REGISTER (grid-stride)
    float v = 0.f;
    for (int i = blockIdx.x * blockDim.x + threadIdx.x; i < n;
         i += gridDim.x * blockDim.x)
        v += in[i];
    s[threadIdx.x] = v;
    __syncthreads();
    // 2) shared-memory tree down to one warp (stride halving keeps
    //    active threads CONTIGUOUS -> no divergence inside a warp)
    for (int stride = 128; stride > 32; stride >>= 1) {
        if (threadIdx.x < stride) s[threadIdx.x] += s[threadIdx.x + stride];
        __syncthreads();
    }
    // 3) last 64 values: finish inside one warp, barrier-free
    if (threadIdx.x < 32) {
        v = s[threadIdx.x] + s[threadIdx.x + 32];
        for (int off = 16; off > 0; off >>= 1)
            v += __shfl_down_sync(0xffffffff, v, off);
        if (threadIdx.x == 0) atomicAdd(out, v);   // 4) one atomic per block
    }
}
// host: cudaMemset(out, 0, 4); for n==0 stop there, otherwise launch
//       with exactly 256 threads per block.
  • The layered structure is the answer: registers (free) → shared tree (cheap) → shuffles (barrier-free) → one atomic per block (expensive, done once). Each layer uses the cheapest mechanism that still works.
  • Follow-up "why stride halving downward?": active threads stay contiguous (0..stride), so whole warps retire; the naive interleaved version (tid % (2*stride) == 0) diverges every warp.
  • Follow-up "alternatives to the atomic?": write per-block partials and run a second small kernel — deterministic order (bitwise-reproducible), vs atomicAdd's run-to-run float variation. Interviewers love that you know reductions reorder math.
  • Production answer: cub::DeviceReduce::Sum — say it after, not instead of, writing the kernel.

4. Matrix transpose (and its twin: rotate an image 90°)

Core

Tests: the read-xor-write coalescing dilemma, shared-memory staging, bank-conflict padding.

Transpose an n×n float matrix. The naive version is trivially correct — why is it slow, and how do you fix it? (The reported NVIDIA "rotate an image 90°" question is this plus one index flip.)

Acceptance tests: launch dim3(32,32); n ∈ {0,1,31,32,33,63,64,65}; unique coordinate-encoded values; transpose twice and compare bitwise; canaries and sanitizer catch edge-tile writes.

Show solution & discussion
__global__ void transpose(const float* __restrict__ in,
                          float* __restrict__ out, int n) {
    __shared__ float tile[32][33];                 // +1: kills bank conflicts
    int x = blockIdx.x * 32 + threadIdx.x;         // read coords
    int y = blockIdx.y * 32 + threadIdx.y;
    if (x < n && y < n)
        tile[threadIdx.y][threadIdx.x] = in[y * n + x];   // coalesced read
    __syncthreads();
    x = blockIdx.y * 32 + threadIdx.x;             // write coords: SWAP the
    y = blockIdx.x * 32 + threadIdx.y;             // BLOCK indices...
    if (x < n && y < n)
        out[y * n + x] = tile[threadIdx.x][threadIdx.y];  // ...and read the
}                                                  // tile transposed: write
                                                   // is coalesced too
// rotate 90° clockwise instead: out is n×n with
//   out[c][n-1-r] = in[r][c]  ->  same tile staging, write column mirrored
  • The dilemma to articulate: direct transpose makes exactly one of read/write strided — a warp's stores scatter across n-float strides, ~32 sectors per request, ~8× write traffic. No index cleverness fixes both sides in global memory.
  • The fix: stage a 32×32 tile in shared memory; global read AND write are both row-contiguous, and the transposition happens in the tile indices where "strided" access is nearly free.
  • Why [32][33]: reading tile[threadIdx.x][…] walks a column; with stride 32 all lanes hit one bank (32-way conflict). Stride 33 → one bank per row. Expect this exact follow-up.
  • Verification answer: transpose twice, compare to the input bitwise (no FP reordering here — memcmp is legal for a transpose).

5. Histogram (256 bins)

Core

Tests: atomics, contention reasoning, the privatization pattern.

Count byte-value frequencies over n bytes into 256 bins. The one-line atomicAdd version works — why does it crawl on skewed data, and what's the standard fix?

Acceptance tests: n ∈ {0,1,255,256,257,1M}; all-one-bin, uniform, and random distributions; zero bins before launch; require exact bins and Σbins == n.

Show solution & discussion
__global__ void hist256(const unsigned char* __restrict__ data, size_t n,
                        unsigned int* __restrict__ bins) {
    __shared__ unsigned int local[256];
    // 1) zero the block-private histogram (cooperatively)
    for (int i = threadIdx.x; i < 256; i += blockDim.x) local[i] = 0;
    __syncthreads();
    // 2) accumulate into SHARED-memory bins: collisions are block-local
    for (size_t i = blockIdx.x * (size_t)blockDim.x + threadIdx.x; i < n;
         i += (size_t)gridDim.x * blockDim.x)
        atomicAdd(&local[data[i]], 1u);
    __syncthreads();
    // 3) merge: 256 global atomics per BLOCK, not per element
    for (int i = threadIdx.x; i < 256; i += blockDim.x)
        atomicAdd(&bins[i], local[i]);
}
  • Why the naive version dies: atomics to one address serialize; a skewed input (an image that's mostly sky) funnels millions of updates into a few global addresses — a queue at L2, not parallel work.
  • Privatization economics: collisions still happen in shared memory but cost ~nanoseconds there; the global merge is 256 atomics per block instead of one per input byte — a ~10,000× reduction in global atomic traffic for large inputs.
  • Follow-ups: byte reads are still coalesced (consecutive lanes read consecutive bytes); size_t indexing matters beyond 2³¹ elements; for few-bin float histograms mention per-warp copies to cut shared-memory contention further.

6. Tiled matrix multiply

Core

Tests: the data-reuse argument (arithmetic intensity), double-barrier discipline, boundary handling.

Write C = A×B for N×N floats using shared-memory tiles, correct for N not a multiple of the tile size. Be ready for: "why is this faster than the naive version — with numbers?"

Acceptance tests: for T ∈ {16,32}, require T² within the device block limit and launch block=(T,T) only when N>0; test N ∈ {0,1,T−1,T,T+1,2T+1} with zero, identity, and random matrices; compare to an FP64 CPU reference and run sanitizer on partial tiles.

Show solution & discussion
template <int T>                                    // T = 16 or 32
__global__ void sgemm(const float* __restrict__ A, const float* __restrict__ B,
                      float* __restrict__ C, int N) {
    __shared__ float As[T][T], Bs[T][T + 1];
    int row = blockIdx.y * T + threadIdx.y;
    int col = blockIdx.x * T + threadIdx.x;
    float acc = 0.f;
    for (int k0 = 0; k0 < N; k0 += T) {
        // guarded, coalesced tile loads (zero-pad outside the matrix)
        As[threadIdx.y][threadIdx.x] =
            (row < N && k0 + threadIdx.x < N) ? A[row * N + k0 + threadIdx.x] : 0.f;
        Bs[threadIdx.y][threadIdx.x] =
            (k0 + threadIdx.y < N && col < N) ? B[(k0 + threadIdx.y) * N + col] : 0.f;
        __syncthreads();                            // tile fully loaded
        #pragma unroll
        for (int k = 0; k < T; ++k)
            acc += As[threadIdx.y][k] * Bs[k][threadIdx.x];
        __syncthreads();                            // done reading before reload
    }
    if (row < N && col < N) C[row * N + col] = acc;
}
// host: if (N > 0), require T*T <= maxThreadsPerBlock and launch
// dim3 block(T,T), grid((N+T-1)/T, (N+T-1)/T); otherwise no launch.
  • The numbers they want: naive matmul loads each A/B element N times from DRAM (~0.25 FLOP/byte — memory-bound). Tiling loads each element once per tile: T× fewer bytes, arithmetic intensity ~T/4 FLOP/byte — at T=32 the kernel crosses to compute-bound.
  • Both barriers are load-bearing: the first prevents computing on a half-loaded tile, the second prevents a fast warp from overwriting a tile others still read. Removing the second is the classic intermittent-corruption bug.
  • Zero-padding the guards keeps the inner loop branch-free — cleaner than guarding the accumulation.
  • Escalation path if asked "faster?": thread coarsening (each thread computes a register tile), vectorized loads, double buffering with cp.async, tensor cores — in that order. And cuBLAS is the production answer.

7. 1-D convolution / stencil with halo

Core

Tests: halo loading, constant memory, boundary handling — the pattern behind all stencil codes.

Apply a 7-tap filter (radius 3) to n floats: out[i] = Σ w[k]·in[i+k], k ∈ [−3, 3], zero beyond the edges. Use shared memory so each input is read from global memory once per block.

Acceptance tests: launch exactly 256 threads; n ∈ {0,1,2,3,255,256,257}; impulse and all-ones inputs; asymmetric weights expose flipped indexing; compare both boundaries and interior against a CPU reference.

Show solution & discussion
__constant__ float c_w[7];                          // broadcast-friendly

__global__ void conv1d_r3(const float* __restrict__ in,
                          float* __restrict__ out, int n) {
    constexpr int R = 3, B = 256;
    __shared__ float s[B + 2 * R];                  // center + two halos
    int base = blockIdx.x * B;
    int g    = base + threadIdx.x;
    s[threadIdx.x + R] = (g < n) ? in[g] : 0.f;     // center (coalesced)
    if (threadIdx.x < R) {
        int lg = base - R + threadIdx.x;            // left halo
        s[threadIdx.x] = (lg >= 0) ? in[lg] : 0.f;
        int rg = base + B + threadIdx.x;            // right halo
        s[B + R + threadIdx.x] = (rg < n) ? in[rg] : 0.f;
    }
    __syncthreads();
    if (g < n) {
        float acc = 0.f;
        #pragma unroll
        for (int k = -R; k <= R; ++k)
            acc += c_w[k + R] * s[threadIdx.x + R + k];
        out[g] = acc;
    }
}
// host: cudaMemcpyToSymbol(c_w, w, sizeof(w));
  • Why shared memory: without it each input element is read up to 7 times from global; with the halo tile, once per block. Same reuse argument as matmul, smaller scale.
  • Why constant memory for weights: every thread reads the same w[k] at the same time — the broadcast case constant cache is built for. (For large/dynamic filters, pass a pointer and rely on L1/read-only cache instead.)
  • Halo subtlety: the first R threads load both halos — fine for R ≤ blockDim; say so. The guards implement zero-padding at array ends.
  • 2-D follow-up: same idea with a (B+2R)² tile — or separable filters as two 1-D passes, which is usually the right answer and worth volunteering.

8. Block-wide inclusive prefix sum (scan)

Hard

Tests: shuffle-based warp scan, hierarchical composition (warp → block → grid).

Compute the inclusive prefix sum of one block's worth of data (e.g. 1024 floats) using warp shuffles. Then explain how you'd extend it to arbitrary n.

Acceptance tests: launch a warp-multiple block of at most 1024 threads; test valid counts {1,31,32,33,1023,1024}, all ones and mixed signs; compare every prefix, not only the final total.

Show solution & discussion
__global__ void block_scan_inclusive(const float* __restrict__ in,
                                     float* __restrict__ out, int n) {
    __shared__ float warp_sums[32];
    int g    = blockIdx.x * blockDim.x + threadIdx.x;
    int lane = threadIdx.x & 31, warp = threadIdx.x >> 5;
    float v = (g < n) ? in[g] : 0.f;

    // 1) inclusive scan WITHIN each warp (Hillis-Steele on shuffles)
    for (int off = 1; off < 32; off <<= 1) {
        float u = __shfl_up_sync(0xffffffff, v, off);
        if (lane >= off) v += u;
    }
    if (lane == 31) warp_sums[warp] = v;            // each warp's total
    __syncthreads();

    // 2) warp 0 scans the 32 warp totals (same trick, one warp's worth)
    if (warp == 0) {
        float w = (lane < blockDim.x / 32) ? warp_sums[lane] : 0.f;
        for (int off = 1; off < 32; off <<= 1) {
            float u = __shfl_up_sync(0xffffffff, w, off);
            if (lane >= off) w += u;
        }
        warp_sums[lane] = w;
    }
    __syncthreads();

    // 3) add the exclusive prefix of your warp's total
    if (warp > 0) v += warp_sums[warp - 1];
    if (g < n) out[g] = v;
}
  • The composition is the answer: scan within warps (no barriers, shuffles), scan the warp totals, add the prefix back. The same three-level shape extends to the grid: per-block sums → scan of block sums (tiny second kernel) → uniform add third kernel. Or a decoupled-lookback single pass, which is what CUB does — name it.
  • Why __shfl_up_sync: lane i needs the value from lane i−off; the lane >= off guard keeps low lanes' values unchanged, which is exactly Hillis-Steele.
  • Scan matters because it looks serial — the interviewer is checking you can parallelize a dependency chain. Stream compaction, radix sort, and memory allocation all sit on top of it.

9. Row-wise numerically stable softmax

Hard — AI-lab favorite

Tests: numerical stability, multi-phase block cooperation, barrier-and-broadcast discipline.

For a rows × cols matrix, compute softmax along each row: y = exp(x − max(x)) / Σexp(x − max(x)). One block per row. Why the max subtraction, and where are the sync hazards?

Acceptance tests: cols ∈ {1,31,32,255,256,257,1024}; constant rows and logits near ±1000; outputs are finite/nonnegative, each row sums near 1, and matches a stable FP64 reference.

Show solution & discussion
__global__ void softmax_rows(const float* __restrict__ X,
                             float* __restrict__ Y, int cols) {
    extern __shared__ float s[];                    // blockDim.x floats
    const float* x = X + (size_t)blockIdx.x * cols; // this block's row
    float*       y = Y + (size_t)blockIdx.x * cols;

    // phase 1: row max (stability: exp(x - max) can't overflow)
    float m = -INFINITY;
    for (int i = threadIdx.x; i < cols; i += blockDim.x)
        m = fmaxf(m, x[i]);
    s[threadIdx.x] = m; __syncthreads();
    for (int st = blockDim.x / 2; st > 0; st >>= 1) {
        if (threadIdx.x < st) s[threadIdx.x] = fmaxf(s[threadIdx.x], s[threadIdx.x + st]);
        __syncthreads();
    }
    m = s[0];
    __syncthreads();          // EVERYONE has read s[0] before phase 2 reuses s

    // phase 2: exponentiate (stash in y) and sum
    float sum = 0.f;
    for (int i = threadIdx.x; i < cols; i += blockDim.x) {
        float e = __expf(x[i] - m);
        y[i] = e;
        sum += e;
    }
    s[threadIdx.x] = sum; __syncthreads();
    for (int st = blockDim.x / 2; st > 0; st >>= 1) {
        if (threadIdx.x < st) s[threadIdx.x] += s[threadIdx.x + st];
        __syncthreads();
    }
    float inv = 1.f / s[0];

    // phase 3: normalize in place
    for (int i = threadIdx.x; i < cols; i += blockDim.x) y[i] *= inv;
}
// launch: softmax_rows<<<rows, 256, 256 * sizeof(float)>>>(X, Y, cols);
  • Why subtract the max: exp(1000) overflows to inf and the row becomes NaN; exp(x − max) ≤ 1 always. The result is mathematically identical — this is THE canonical numerical-stability question.
  • The subtle barrier: the __syncthreads() after m = s[0] — without it a fast thread starts phase 2 and overwrites s[0] while slow threads are still reading the max. Interviewers plant exactly this hazard.
  • Traffic accounting: this implementation reads x once for max and again for exp, then writes and rereads y for normalization: 3 global reads + 2 writes per element. Fusing with a following op or keeping a small row in registers/shared reduces that traffic. Online softmax (the FlashAttention building block) combines running max and sum without materializing the score matrix.

10. Matrix–vector multiply, one warp per row

Hard

Tests: choosing the work-to-thread mapping, warp-level reduction, coalescing within a row.

Compute y = A·x for an N×M matrix. One thread per row is the obvious mapping — why is it wrong, and what's better?

Acceptance tests: launch 256-thread blocks only when N>0; N ∈ {0,1,7,8,9}, M ∈ {0,1,31,32,33,1025}; zero and random matrices; FP64 dot-product reference with tolerance; canaries cover the final partial warp.

Show solution & discussion
__global__ void gemv_warp_per_row(const float* __restrict__ A,
                                  const float* __restrict__ x,
                                  float* __restrict__ y, int N, int M) {
    int row  = (blockIdx.x * blockDim.x + threadIdx.x) / 32;
    int lane = threadIdx.x & 31;
    if (row >= N) return;                 // whole warp exits together
    if (M == 0) {                          // avoid arithmetic on a null A
        if (lane == 0) y[row] = 0.f;
        return;                            // uniform for the whole row warp
    }
    const float* a = A + (size_t)row * M;
    float acc = 0.f;
    for (int j = lane; j < M; j += 32)    // lanes read CONSECUTIVE floats
        acc += a[j] * x[j];               // -> coalesced
    for (int off = 16; off > 0; off >>= 1)
        acc += __shfl_down_sync(0xffffffff, acc, off);
    if (lane == 0) y[row] = acc;
}
// host: if (N > 0), launch 256-thread blocks -> 8 rows/block,
//       grid = ceil(N / 8); otherwise there is no output work.
  • Why one thread per row fails: thread i would read A[i*M + j] — warp-neighbor threads are M floats apart, fully strided access, ~32× the traffic. The mapping, not the math, is the whole question.
  • Warp-per-row fixes it: lanes stride through one row together, consecutive addresses, coalesced; the dot product finishes with a shuffle tree. For very wide rows use a block per row; for many short rows, a thread per row with A stored column-major. "It depends on M — here's the crossover reasoning" is the senior answer.
  • The early return is safe because the division makes the whole warp share one row — no divergent exit. Worth saying aloud.

11. Find the three bugs

Debug round

Tests: race detection, boundary reasoning — reviewing code you didn't write, exactly like the job.

This kernel should compute a 3-point moving average with clamped edges and is launched with exactly 256 threads per block: out[i] = (in[i−1] + in[i] + in[i+1]) / 3. It compiles, runs, and is wrong three different ways. Find all three.

Acceptance tests: n ∈ {0,1,255,256,257,511,512} with coordinate-encoded values; exact edge semantics; memcheck and racecheck both clean over repeated launches.

__global__ void smooth(const float* in, float* out, int n) {   // BUGGY
    __shared__ float s[256];
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    s[threadIdx.x] = in[i];
    if (threadIdx.x > 0 && threadIdx.x < blockDim.x - 1)
        out[i] = (s[threadIdx.x - 1] + s[threadIdx.x] + s[threadIdx.x + 1]) / 3.f;
}
Show the three bugs & the fixed kernel
  • Bug 1 — out-of-bounds read: in[i] has no i < n guard; the last block reads (and the compute line writes) past the array.
  • Bug 2 — race on shared memory: no __syncthreads() between the tile store and the neighbor reads. A thread can read s[threadIdx.x ± 1] before the neighboring warp has written it.
  • Bug 3 — holes in the output: the edge threads of every block (threadIdx.x == 0 and blockDim.x − 1) never write out[i] — two unwritten elements per block interior. The block needs halo data, not fewer writers.
__global__ void smooth_fixed(const float* __restrict__ in,
                             float* __restrict__ out, int n) {
    __shared__ float s[256 + 2];                    // + halo cells
    if (n == 0) return;                             // uniform block-wide exit
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    int c = threadIdx.x + 1;
    // Invalid lanes in a partial block load the clamped endpoint. The lane
    // after i==n-1 then supplies the right neighbor with no overlapping writer.
    s[c] = in[min(i, n - 1)];                                  // bug 1 fixed
    if (threadIdx.x == 0)                // left/right halo, edge-clamped
        s[0]   = (i > 0)                 ? in[i - 1]              : in[0];
    if (threadIdx.x == blockDim.x - 1)
        s[c+1] = in[min(i + 1, n - 1)];
    __syncthreads();                                          // bug 2 fixed
    if (i < n)                                                // bug 3 fixed:
        out[i] = (s[c - 1] + s[c] + s[c + 1]) / 3.f;          // all threads write
}
  • Interview meta: narrate a systematic sweep — bounds first, then every shared write→read edge needs a barrier, then "does every output element have exactly one writer?" That checklist finds all three without luck. Use Compute Sanitizer memcheck for the out-of-bounds access and racecheck for the shared-memory hazard.

12. Host-side: pipeline 8 batches with streams

Host round

Tests: pinned memory, stream semantics, double-buffered device memory, correct timing.

You must process 8 independent batches: upload → kernel → download each. Write host code that overlaps transfers with compute, and time the whole pipeline correctly.

Acceptance tests: compare all 8 outputs with a serial pipeline; use tiny and tail-sized batches; verify event timing against a synchronized CPU timer; nsys must show only hardware-supported overlap and no buffer reuse race.

Show solution & discussion
const int  NB = 8;
const size_t bytes = BATCH * sizeof(float);

float *h_in, *h_out;
cudaMallocHost(&h_in,  NB * bytes);      // pinned for reliable async overlap
cudaMallocHost(&h_out, NB * bytes);

float *d_in[2], *d_out[2];               // double-buffered device memory:
for (int i = 0; i < 2; ++i) {            // stream k uses buffer k % 2
    cudaMalloc(&d_in[i],  bytes);
    cudaMalloc(&d_out[i], bytes);
}
cudaStream_t st[2];
cudaStreamCreate(&st[0]); cudaStreamCreate(&st[1]);

cudaStream_t timing;
cudaStreamCreateWithFlags(&timing, cudaStreamNonBlocking);
cudaEvent_t t0, t1, done[2];
cudaEventCreate(&t0); cudaEventCreate(&t1);
cudaEventCreateWithFlags(&done[0], cudaEventDisableTiming);
cudaEventCreateWithFlags(&done[1], cudaEventDisableTiming);
cudaEventRecord(t0, timing);
cudaStreamWaitEvent(st[0], t0);           // common start, explicit in every mode
cudaStreamWaitEvent(st[1], t0);

for (int b = 0; b < NB; ++b) {
    int k = b & 1;                        // alternate stream + buffers
    cudaMemcpyAsync(d_in[k], h_in + b * BATCH, bytes,
                    cudaMemcpyHostToDevice, st[k]);
    process<<<grid, block, 0, st[k]>>>(d_in[k], d_out[k], BATCH);
    cudaMemcpyAsync(h_out + b * BATCH, d_out[k], bytes,
                    cudaMemcpyDeviceToHost, st[k]);
}
cudaEventRecord(done[0], st[0]);
cudaEventRecord(done[1], st[1]);
cudaStreamWaitEvent(timing, done[0]);      // explicitly join both worker streams
cudaStreamWaitEvent(timing, done[1]);
cudaEventRecord(t1, timing);
cudaEventSynchronize(t1);
float ms; cudaEventElapsedTime(&ms, t0, t1);
  • Why buffer reuse is safe: batch b+2 reuses batch b's device buffers, but both are queued on the same stream, and streams execute in order — the copy for b+2 cannot start until b's work drained. Cross-batch safety comes free from stream ordering; say this explicitly.
  • Requirements for overlap: use pinned host memory for reliable asynchronous DMA, distinct streams, independent device buffers, and hardware that supports the desired transfer/compute concurrency. Query device capabilities such as asyncEngineCount; do not assume a fixed number of engines.
  • Timing correctness: the timing stream explicitly waits for completion events from both workers, so this works with legacy and per-thread default-stream semantics and with non-blocking streams. A CPU timer is also valid if it synchronizes both workers before stopping.
  • Expected speedup: throughput is bounded by the slowest pipeline stage and by actual copy-engine concurrency. Verify in nsys—the timeline should show the intended overlap instead of trusting an idealized 2–3× estimate.

13. Stable stream compaction (copy_if)

Data pipeline

Tests: multi-phase algorithm design, scan, output sizing, and knowing when to use CUB.

Given n floats, copy values strictly greater than zero to out, preserve their input order, and return the selected count in device memory. Handle n = 0, arbitrary tails, negative zero, and NaNs (both are not selected). Give the production solution, then sketch the custom algorithm.

Acceptance tests: none/all/alternating selected; n ∈ {0,1,31,32,33,1M}; include −0, +0, NaN, and duplicates; require exact count, stable order, and untouched output beyond the count.

Show solution & discussion
#include <cub/cub.cuh>

struct IsPositive {
    __host__ __device__ bool operator()(float x) const { return x > 0.f; }
};

void compact_positive(const float* d_in, float* d_out, int* d_count,
                      int n, cudaStream_t stream) {
    if (n == 0) {
        cudaMemsetAsync(d_count, 0, sizeof(*d_count), stream);
        return;
    }
    void* d_temp = nullptr;
    size_t temp_bytes = 0;
    cub::DeviceSelect::If(d_temp, temp_bytes, d_in, d_out, d_count,
                          n, IsPositive{}, stream);   // storage query
    cudaMallocAsync(&d_temp, temp_bytes, stream);
    cub::DeviceSelect::If(d_temp, temp_bytes, d_in, d_out, d_count,
                          n, IsPositive{}, stream);   // stable selection
    cudaFreeAsync(d_temp, stream);                    // ordered after use
}
  • Custom shape: kernel 1 writes a 0/1 predicate per item; device-wide exclusive scan produces each selected item's stable destination; kernel 2 scatters selected values. The count is prefix[n-1] + flag[n-1], with a separate n == 0 path.
  • Why not one atomic per winner? a global counter gives a fast unordered compaction but arrival order is not input order. Warp-aggregated reservations reduce atomics when stability is not required.
  • Production judgment: CUB already handles tails, tuning, and architecture differences. A custom fused version is justified when the predicate is part of a larger pipeline and deleting intermediate flags/traffic matters.
  • Reference: CUB device-wide primitives.

14. CSR sparse matrix–vector multiply

Sparse / ragged

Tests: irregular work mapping, gathers, warp reduction, and load-balance judgment.

Compute y = A·x where A is CSR: row_offsets[rows+1], col_indices[nnz], and values[nnz]. Rows may be empty or highly skewed. Start with one warp per row; explain when it fails and how production code handles the distribution.

Acceptance tests: zero-nnz and empty rows, duplicate columns, rows of length {1,31,32,33,10K}, and skewed mixtures; upstream validation rejects invalid offsets/indices before launch; compare to a CPU CSR reference with tolerance.

Show solution & discussion
__global__ void csr_spmv_warp(const int* __restrict__ row_offsets,
                              const int* __restrict__ col_indices,
                              const float* __restrict__ values,
                              const float* __restrict__ x,
                              float* __restrict__ y, int rows) {
    int thread = blockIdx.x * blockDim.x + threadIdx.x;
    int row     = thread >> 5;
    int lane    = threadIdx.x & 31;
    if (row >= rows) return;                         // whole warp exits

    int begin = row_offsets[row], end = row_offsets[row + 1];
    float sum = 0.f;
    for (int j = begin + lane; j < end; j += 32)
        sum = fmaf(values[j], x[col_indices[j]], sum);
    for (int off = 16; off > 0; off >>= 1)
        sum += __shfl_down_sync(0xffffffff, sum, off);
    if (lane == 0) y[row] = sum;                    // empty row writes 0
}
// launch with a multiple-of-32 block size, e.g. 256 threads = 8 rows/block
  • What coalesces: lanes traverse consecutive values and col_indices; x[col] is an irregular gather whose cache behavior depends on column locality.
  • Where it breaks down: one warp wastes lanes on very short rows and leaves a single warp running for extremely long rows. Bucket rows by length, use threads/warp/block per row adaptively, or partition long rows into mergeable chunks.
  • Production answer: benchmark cuSPARSE across the real matrix family. Validate empty rows, duplicate columns, invalid indices, and tolerance; floating reduction order need not be bitwise identical to a CPU loop.
  • Reference: cuSPARSE documentation and CSR format.

15. Fused residual + RMSNorm

AI kernel

Tests: mixed-precision accumulation, fusion, two-phase block cooperation, and tail handling.

For each [row, H] finite FP16 vector, compute v = x + residual and y[j] = v[j] · rsqrt(mean(v²) + eps) · gamma[j]. Use finite FP16 gamma, one 256-thread block per row, H>0, and eps>0; require the FP32 reference result to fit finite FP16 (otherwise define saturation or use FP32 output). Accumulate the reduction in FP32 and explain the extra memory pass and how LayerNorm would differ.

Acceptance tests: H ∈ {1,31,32,255,256,257,4096}; zero, constant, and large-magnitude finite rows with FP16-representable reference outputs; compare against an FP32 reference, require finite outputs, and verify the residual is added exactly once.

Show solution & discussion
#include <cuda_fp16.h>

__global__ void residual_rmsnorm(const half* __restrict__ x,
                                 const half* __restrict__ residual,
                                 const half* __restrict__ gamma,
                                 half* __restrict__ y, int H, float eps) {
    __shared__ float partial[256];                   // launch exactly 256
    size_t base = (size_t)blockIdx.x * H;
    float sumsq = 0.f;
    for (int j = threadIdx.x; j < H; j += blockDim.x) {
        float v = __half2float(x[base + j]) + __half2float(residual[base + j]);
        sumsq = fmaf(v, v, sumsq);
    }
    partial[threadIdx.x] = sumsq;
    __syncthreads();
    for (int stride = 128; stride > 0; stride >>= 1) {
        if (threadIdx.x < stride)
            partial[threadIdx.x] += partial[threadIdx.x + stride];
        __syncthreads();
    }
    float inv_rms = rsqrtf(partial[0] / (float)H + eps);
    for (int j = threadIdx.x; j < H; j += blockDim.x) {
        float v = __half2float(x[base + j]) + __half2float(residual[base + j]);
        float g = __half2float(gamma[j]);
        y[base + j] = __float2half_rn(v * inv_rms * g);
    }
}
  • Why FP32: summing FP16 squares over a wide row loses precision or overflows quickly. Inputs/outputs can stay FP16 while reductions and scaling use FP32.
  • Traffic tradeoff: values are reread after the reduction because arbitrary H may not fit in registers/shared memory. For small fixed H, cache v; for large H, the second coalesced pass is often the right occupancy trade.
  • LayerNorm follow-up: RMSNorm does not subtract a mean. LayerNorm needs stable mean and variance; parallel Welford/Chan states are stronger than E[x²] − E[x]² for large, nearly equal values.
  • Reference: cuDNN normalization operations.

16. Dynamic per-row INT8 quantization

Inference

Tests: reduction + transform fusion, numerical contracts, rounding, saturation, and degenerate rows.

Quantize each finite FP32 row symmetrically to signed INT8 using one 256-thread block per row. Write one FP32 scale per row such that x ≈ q·scale, use the chosen symmetric range [-127, 127], round to nearest, clamp, and define all-zero rows. Support arbitrary positive cols; reject or sanitize NaN/Inf before this kernel.

Acceptance tests: cols ∈ {1,31,32,255,256,257}; all-zero, ±max, and rounding-boundary values; require q in [-127,127], zero-row scale 1, and dequantization error near ≤0.5·scale.

Show solution & discussion
#include <cfloat>

__global__ void quantize_rows_i8(const float* __restrict__ x,
                                 signed char* __restrict__ q,
                                 float* __restrict__ scales, int cols) {
    __shared__ float smax[256];                      // launch exactly 256
    size_t base = (size_t)blockIdx.x * cols;
    float m = 0.f;
    for (int j = threadIdx.x; j < cols; j += blockDim.x)
        m = fmaxf(m, fabsf(x[base + j]));
    smax[threadIdx.x] = m;
    __syncthreads();
    for (int stride = 128; stride > 0; stride >>= 1) {
        if (threadIdx.x < stride)
            smax[threadIdx.x] = fmaxf(smax[threadIdx.x], smax[threadIdx.x + stride]);
        __syncthreads();
    }
    float maxabs = smax[0];
    // FLT_MIN keeps a nonzero representable scale for subnormal-only rows.
    float scale = (maxabs > 0.f) ? fmaxf(maxabs / 127.f, FLT_MIN) : 1.f;
    if (threadIdx.x == 0) scales[blockIdx.x] = scale;
    for (int j = threadIdx.x; j < cols; j += blockDim.x) {
        int v = __float2int_rn(x[base + j] / scale); // nearest-even
        v = max(-127, min(127, v));
        q[base + j] = (signed char)v;
    }
}
  • Contract details matter: signed INT8's full range is [-128,127], while deliberately choosing the symmetric subset [-127,127] makes the scale simple. An all-zero row maps to all zeros with scale 1; flooring a nonzero scale at FLT_MIN keeps subnormal-only rows defined. This kernel's finite-input precondition must be enforced upstream because NaN/Inf do not have a meaningful quantized result.
  • Why two passes: the scale is unknown until the maximum reduction completes. Cache small rows or reread large rows; vectorized loads/stores need aligned bodies plus scalar tails.
  • Follow-up: per-row scaling preserves outlier-sensitive rows better than one scale for the whole tensor, while finer per-block scales improve accuracy at the cost of scale storage and more complex consumers. Compare dequantized error, not only kernel time.
  • Reference: TensorRT quantization schemes.
