Task 1 · Profiler judgment

Triage inefficient memory access

10 min12 points

A single FP32 gather kernel takes 2.40 ms on a fixed large input. Nsight Compute reports:

SignalObservedInterpretation constraint
DRAM throughput38% of peakNot near the device bandwidth ceiling
L2 hit rate81%A high hit rate alone does not mean accesses are efficient
L1/TEX sectors / request15.2Ideal is 4 for 32 active lanes loading one aligned FP32 value each
Long scoreboard stalls54% of issue stallsConsistent with waiting on data, but not proof of root cause
Achieved occupancy61%Reasonable; raising it is not automatically useful

Deliverable: classify the leading bottleneck hypothesis; identify the first source/index-layout change you would test; name the before/after metrics and correctness checks; give one experiment that could falsify your hypothesis; explain why L2 hit rate and occupancy alone do not justify the change.

Task 2 · Correctness review

Debug an unfamiliar reduction

15 min18 points

The caller expects one scalar sum for any n >= 0. It launches multiple blocks and initializes *out = 0.

__global__ void reduce_bad(const float* x, float* out, int n) {
  extern __shared__ float s[];
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i >= n) return;
  s[threadIdx.x] = x[i];
  __syncthreads();

  for (int stride = blockDim.x / 2; stride > 0; stride >>= 1) {
    if (threadIdx.x < stride) s[threadIdx.x] += s[threadIdx.x + stride];
    __syncthreads();
  }
  if (threadIdx.x == 0) *out = s[0];
}

Deliverable: provide a safe correction and launch/host contract. Cover partial blocks, arbitrary positive n, cross-block combination, n = 0, output initialization/ownership, and the synchronization rule violated by the original. You may use an atomic final combine or a second reduction pass—state the numerical and performance tradeoff.

Task 3 · Kernel implementation

Implement ragged stable softmax

30 min32 points
__global__ void ragged_softmax(
    const float* x, float* y,
    const int* lengths, int rows, int stride);

Use exactly one 256-thread block per row. Upstream validation guarantees rows >= 0, 1 <= stride <= 4096, 0 <= lengths[r] <= stride, and finite FP32 inputs in each valid prefix. For every row, write stable softmax over [0, length) and write zero to all padding. An empty row is all zeros; a zero-row workload is a host-side no-op.

Acceptance criteria: safe for lengths {0,1,31,32,255,256,257,4096}; handles arbitrary tails without divergent barriers; subtracts a block-wide maximum; uses valid cross-warp/block reduction; writes every padded element; emits finite nonnegative results; matches a double-precision CPU reference with each nonempty row sum within 2e-5. Include launch and error-checking notes.

Task 4 · Optimization tradeoff

Explain a fusion regression

12 min14 points

Two individually optimized kernels were fused. Correctness is unchanged on the tested shapes, but the representative steady-state result regressed:

SignalBefore fusionAfter fusion
End-to-end GPU time1.35 ms1.88 ms
Registers / thread52148
Theoretical occupancy62.5%18.75%
Local memory traffic0 B96 B / thread
Removed intermediate traffic0640 MB

Deliverable: explain the likely causal chain without claiming that occupancy alone caused the regression. Propose the next controlled experiment, the source-level evidence and counters you would inspect, and at least two possible remedies. State when partial fusion or keeping both kernels is the correct engineering decision.

Task 5 · Host orchestration

Repair streams, dependencies, and timing

13 min14 points

A loop intended to overlap work uses two streams created with cudaStreamNonBlocking but a single device buffer. The host input/output came from malloc, so the program cannot rely on true transfer overlap. It nevertheless treats every call as asynchronous: each iteration issues cudaMemcpyAsync(H2D), a kernel, and cudaMemcpyAsync(D2H), alternating streams. The next stream overwrites the same device buffer without waiting, and the CPU reads output without an explicit completion check. Start/stop events are recorded on the default stream around the loop.

Deliverable: sketch corrected host pseudocode. Address pinned memory, buffer ownership or explicit cross-stream events, host buffer lifetime, completion before CPU consumption, event placement that actually joins all non-blocking streams, warm-up, and error checking. The repair must remain correct after pageable buffers are replaced with pinned storage; do not depend on implementation-specific staging or blocking for ordering. Explain what can overlap and what must remain ordered.

Task 6 · Performance communication

Defend—or reject—a 4.1× speedup claim

10 min10 points

A teammate reports “4.1× faster.” The baseline is a debug build; the candidate is optimized with -O3. The baseline ran once after cold allocation, the candidate ran once after three warm-ups. Inputs differ but have the same element count. There is no CPU reference, synchronization is implicit, clocks/architecture are unreported, and only the fastest timing was kept.

Deliverable (maximum 200 words): decide whether the claim is currently defensible. Define a minimal correctness oracle and fair benchmark protocol, including equivalent builds/work, representative shapes and edge cases, warm-up/cache policy, correct GPU timing, repeated samples and reported statistic, environment disclosure, and profiler evidence that would explain—not merely decorate—the result.
