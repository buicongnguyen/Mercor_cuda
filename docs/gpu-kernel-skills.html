1. SIMT Execution Model & Architecture

The machine model every optimization reasons against: streaming multiprocessors, warps in lockstep, blocks that must be independent, and an SM that hides latency by switching warps instead of predicting branches. Vocabulary is CUDA's; HIP and shader models map one-to-one.

Key terms

SM (streaming multiprocessor)
The GPU's core unit: warp schedulers, ALU/FP pipes, tensor cores, a register file, and L1/shared memory. A kernel's blocks are distributed across SMs; each block runs entirely on one SM.
Warp
A fixed group of 32 threads issued together. The hardware schedules and issues instructions per warp, never per thread. (AMD calls it a wavefront — 64 lanes on CDNA.)
SIMT
Single Instruction, Multiple Threads: one instruction is executed by all active lanes of a warp on different data. Threads may branch independently, paid for with masked serial execution.
Thread block
Up to 1024 threads that share shared memory and can barrier-sync with __syncthreads(). Blocks must not depend on each other's execution order.
Grid
All blocks of one launch. No grid-wide barrier exists inside a kernel; grid-level ordering comes from consecutive kernel launches on a stream (or cooperative groups).
Warp scheduler
Per-SM units (typically 4) that each cycle pick an eligible resident warp and issue its next instruction. Stalled warps cost nothing while others can issue.
Occupancy
Resident warps on an SM divided by the hardware maximum (e.g. 48–64 warps). Limited by registers/thread, shared memory/block, and block-slot count. A means to hide latency, not a score to maximize.
Warp divergence
Threads of one warp taking different branch paths: the warp executes each path serially with non-participating lanes masked. Divergence between different warps is free.
Register file
Per-SM pool (typically 64K 32-bit registers). Each thread's registers are private and allocated for its whole lifetime — heavy usage per thread cuts how many warps fit.
Latency hiding
The GPU's answer to memory latency: instead of caches and out-of-order execution, keep enough warps resident that some warp is always ready to issue while others wait.
Independent thread scheduling
Since Volta, each thread has its own program counter, so lanes of a warp can interleave. Warp-level intrinsics therefore take an explicit mask and a _sync suffix (e.g. __shfl_down_sync(0xffffffff, …)).
Tail effect
End-of-kernel (or end-of-wave) underutilization: the last wave of blocks doesn't fill all SMs, so achieved occupancy and throughput sag even though the kernel is otherwise healthy.
Active vs eligible warp
Active/resident = allocated on the SM; eligible = ready to issue this cycle (not stalled). Schedulers pick among eligible warps — performance is having an eligible warp every cycle, not merely many active ones.
ILP (instruction-level parallelism)
Independent instructions within one thread that overlap in the pipeline. The other latency-hiding currency besides warp count: a thread with 4 independent loads in flight hides as much latency as 4 threads with one each.
Predication
For short branches the compiler emits both sides as per-lane masked instructions instead of a real branch — no divergence penalty, which is why tiny if bodies are often free.
Cooperative groups
API objects for thread groupings (tile, warp, block, grid) each with their own sync() and collectives. Grid-wide sync exists but requires a cooperative launch that guarantees all blocks are co-resident.
Wave
One round of blocks that fit on the whole GPU simultaneously; a grid executes in ceil(blocks ÷ blocks-per-wave) waves. Runtime cliffs at wave boundaries are wave quantization.

Core concepts

Think in warps, not threads

The scheduler issues to 32-thread warps. An if that splits a warp runs both sides serially with lanes masked; a warp requesting 32 scattered addresses pays for 32 memory transactions. Fast kernels arrange work so warp-neighbor threads branch together and touch neighboring data. Corollary: block sizes should be multiples of 32 — a 100-thread block occupies 4 warps and wastes 28 lanes of the last one.

Latency hiding, not latency avoidance

A global load takes hundreds of cycles. CPUs attack that with big caches, prefetchers, and out-of-order engines; a GPU simply switches to another resident warp — a zero-cost context switch, because every warp's registers stay allocated. This works only if enough warps are resident (occupancy) and they have independent work. Both occupancy and per-thread ILP are ways to buy "bytes in flight".

What limits occupancy

Three per-SM budgets: registers (64K × 32-bit — at 128 registers/thread only 512 threads fit), shared memory (a 48 KB/block kernel on a 100 KB SM fits 2 blocks), and block slots (max blocks per SM). The occupancy calculator in Nsight Compute shows which budget binds. Fixes: fewer registers (__launch_bounds__), smaller tiles, or different block sizes — but verify the fix didn't cause register spills.

Blocks are independent by contract

Blocks may run in any order, on any SM, possibly never concurrently. "Block 0 produces, block 1 consumes" is a race or deadlock. Grid-wide agreement = end the kernel and launch another (launches on one stream are ordered), or atomics designed for it. This contract is what lets the same binary scale from 10 SMs to 150.

Divergence: cost and cures

Divergence cost is proportional to how many distinct paths a warp executes. Cures: sort/partition work so warp-mates take the same branch, convert short branches to predication (compilers do this for small ifs automatically), move uniform decisions out of the kernel via template parameters, and remember: a branch on blockIdx or on a value uniform across the warp costs nothing.

Barriers and their scope

__syncthreads() synchronizes one block, and every thread of the block must reach it — calling it inside a divergent branch is undefined behavior (historically a deadlock). Warp-level exchanges don't need it: shuffles are already synchronized by their mask. Cooperative groups generalize the idea (tile, block, grid scopes) with explicit group objects.

2. Memory Hierarchy & Access Patterns

Most kernels are memory-bound, so the core skill is moving fewer bytes and moving them in the shapes the hardware likes: coalesced sectors from DRAM, conflict-free banks in shared memory, and reuse staged at the right level.

Key terms

Global memory
Device DRAM (GDDR/HBM), visible to all threads, hundreds of GB/s to TB/s of bandwidth but ~400–800 cycles of latency. The usual bottleneck.
Sector
The 32-byte unit the memory system actually moves. A 128-byte cache line is four sectors; the profiler counts sectors, and wasted sectors are wasted bandwidth.
Coalescing
When a warp's 32 addresses fall in few sectors, the hardware merges them: 32 consecutive 4-byte loads = 128 bytes = 4 sectors in one request. Scattered addresses can cost 32 sectors for the same 128 useful bytes.
L1 cache / shared memory
One per-SM physical array split between hardware-managed L1 and programmer-managed shared memory (carve-out configurable). Shared memory ≈ 20–30× lower latency than global.
Shared memory banks
Shared memory is divided into 32 banks, 4 bytes wide. Warp accesses to different banks (or the same address — broadcast) proceed in parallel; different addresses in one bank serialize (bank conflict).
L2 cache
Device-wide cache (tens of MB on modern GPUs) shared by all SMs; every global access that misses L1 goes through it. The coherence point of the device.
Local memory
Per-thread spill space that lives in device memory (cached in L1/L2) despite the name. Register spills and indexed thread-local arrays end up here — visible in ncu as LDL/STL instructions.
Constant memory
Small cached read-only region optimized for broadcast: fastest when every thread of a warp reads the same address; divergent addresses serialize.
Read-only / texture path
__ldg() / const __restrict__ loads route through the read-only data cache; texture objects add addressing/filtering hardware.
Vectorized access
float4/int4 (128-bit) loads move 16 bytes per thread in one instruction — fewer instructions and fewer requests for the same bytes. Requires 16-byte alignment.
Async copy (cp.async / TMA)
Ampere's cp.async copies global→shared without staging through registers; Hopper/Blackwell's TMA engine transfers whole tiles asynchronously, freeing threads and registers for compute.
Pinned (page-locked) host memory
Host allocation that can't be paged out, enabling faster DMA transfers and true overlap of cudaMemcpyAsync with kernels via streams.
__restrict__
Programmer promise that pointers don't alias, freeing the compiler to cache and reorder loads (and route const pointers through the read-only path). A missing restrict is a common reason the compiler re-loads values it "should" have kept.
Unified (managed) memory
cudaMallocManaged: one pointer valid on host and device, migrated on demand via page faults. Convenient; performing well needs cudaMemPrefetchAsync and access hints, or fault storms eat the win.
Zero-copy (mapped) memory
The device dereferences pinned host memory over PCIe on every access. Right for data touched exactly once; disastrous for anything reused — each access repays the full bus latency.
Line vs sector
A 128-byte cache line is four 32-byte sectors. Caches track lines; the memory system moves sectors — profiler traffic counts are in sectors, and unused sector bytes are the waste that coalescing metrics expose.
Alignment
Vector loads require naturally aligned addresses (float4 → 16 bytes). cudaMalloc returns ≥256-byte-aligned pointers; it's the offset arithmetic afterwards (ptr + 3) that silently kills vectorization.

Core concepts

Count sectors, not accesses

The question the hardware asks is not "how many loads?" but "how many 32-byte sectors did those loads touch, and how many of the bytes in each were used?" Warp-consecutive 4-byte loads: 4 sectors, 100% used. Stride-2: 8 sectors, 50% used — 2× traffic for the same data. Fully scattered: up to 32 sectors, ~12% used. Nsight Compute's "sectors per request" makes this visible: ~4 is ideal for 32-bit accesses, 32 is a layout bug.

SoA beats AoS on GPUs

With an array of structs, thread i reading p[i].x strides over the other fields, wasting most of each sector. Structure-of-arrays puts all x together so warp-neighbor threads read memory-neighbor values. Same data, same loads, order-of-magnitude difference in effective bandwidth — the most common single fix in real codebases.

Shared memory is for reuse

Stage a tile of global data into shared memory once, __syncthreads(), then let the whole block reuse it many times — the tiled matrix-multiply pattern that turns a memory-bound loop into a compute-bound one. Two pitfalls: forgetting the second barrier before overwriting the tile, and bank conflicts on column-wise access (fix: pad the tile to [T][T+1] so column neighbors land in different banks).

Registers are the fastest memory — until they spill

The compiler keeps hot values in registers; past ~255 per thread (or a maxrregcount cap) it spills to local memory — device DRAM with a cache in front. A "harmless" occupancy tweak that forces spills can make a kernel slower: watch for LDL/STL in the SASS and the local-memory traffic counters in ncu. Dynamic indexing into a small per-thread array also silently lands in local memory.

The L2 is a bandwidth amplifier

All SMs share L2. When many blocks read the same region (weights, lookup tables, halo cells), L2 serves them at multiples of DRAM bandwidth. Working set slightly bigger than L2? Reorder the schedule (block-swizzling, tiling over the grid) so blocks that share data run near each other in time. Streaming kernels that touch each byte exactly once get nothing from L2 — by design.

Keep the pipeline on the device

PCIe is tens of GB/s; HBM is thousands. Any workflow that copies down between kernels is transfer-bound by design. Upload once, chain kernels on device buffers, download once; use pinned memory + cudaMemcpyAsync + streams to overlap the copies that remain with compute of the previous batch.

3. Profiler-Guided Optimization — Nsight Compute

The heart of the job: run ncu, classify the kernel as compute-, memory-, or latency-bound from speed-of-light and stall data, pick the fix that matches, and know when a metric (L2 hit rate, occupancy) is lying to you — or simply irrelevant.

Key terms

Nsight Systems (nsys)
Whole-application timeline profiler: kernels, memcpys, CPU threads, API calls, gaps. Always the first tool — it tells you which kernel (or copy, or gap) deserves attention.
Nsight Compute (ncu)
Single-kernel profiler: replays a kernel to collect detailed hardware counters, organized into sections (Speed of Light, Memory Workload, Scheduler/Warp Statistics, Occupancy, Source Counters) with rule-based guided analysis.
Speed of Light (SOL)
Headline pair of percentages: SM throughput and memory throughput, each relative to that GPU's peak. The first classification: which resource is closest to its ceiling?
Compute-bound
SM SOL high (~>70–80%): the ALU/tensor pipes are the ceiling. Fixes: cheaper math, lower precision/tensor cores, better algorithm — memory tuning won't help.
Memory-bound
Memory SOL high: some level of the memory system is the ceiling. Drill into Memory Workload Analysis to find which level (DRAM, L2, L1, shared).
Latency-bound
Both SOLs low: nothing saturated — the SM idles between dependent operations. Not enough warps in flight, too little ILP, divergence, or serialization. The subtle case.
Theoretical vs achieved occupancy
Theoretical = best the launch config allows given register/shared budgets. Achieved = measured average. A big gap points at load imbalance, tail effect, or too few blocks — not at resource limits.
L2 hit rate
Fraction of L2 requests served without going to DRAM. High = reuse is being captured; low can be a problem (working set thrashing) or perfectly fine (pure streaming, each byte read once).
L2 / DRAM throughput
Actual bytes/s through each level vs its peak. High L2 + low DRAM throughput = cache is amplifying bandwidth; DRAM near peak = you're paying full price for every byte.
Sectors per request
Average 32-byte sectors moved per warp memory request. ≈4 is ideal for 32-bit accesses; larger means uncoalesced access and wasted bandwidth.
Warp stall reasons
Per-cycle sample of why warps couldn't issue: Long Scoreboard (waiting on global/local memory), Short Scoreboard (shared memory), Barrier, Wait (fixed-latency dependency), MIO/LG Throttle (memory-op queues full), Not Selected (eligible, scheduler picked another — benign).
Roofline / arithmetic intensity
AI = FLOPs per byte of DRAM traffic. Below the ridge point (peak FLOPs ÷ peak bandwidth) a kernel is bandwidth-limited: more FLOP/s is impossible without raising reuse.
Effective bandwidth
(bytes read + written) ÷ kernel time, compared to peak. Two lines of arithmetic that instantly say whether a memory-bound kernel is near the roof or leaving 5× on the table.
Kernel replay
ncu re-runs the kernel several times to collect all counters; results assume repeatable behavior. Profile with locked clocks (--clock-control) for stable numbers.
PTX vs SASS
PTX is the portable virtual ISA; SASS is the real machine code per architecture. ncu's Source page correlates C++ ↔ SASS (build with -lineinfo) — register spills show up there as LDL/STL instructions.
-Xptxas -v
Compile-time report of registers/thread, shared memory, and spill bytes per kernel — the zero-cost first profiler. Check it before and after any register-pressure change.
Little's law (bytes in flight)
Concurrency needed = latency × bandwidth. A 500 ns, 1.6 TB/s memory system needs ~800 KB of requests in flight to saturate — the formal reason occupancy and ILP exist at all.
Issue rate / IPC
Instructions issued per scheduler per cycle. Low IPC with unsaturated pipes corroborates latency-bound; high IPC with busy math pipes corroborates compute-bound.
Baseline diff
ncu stores a profile as a baseline and diffs later runs metric by metric — turning "I think it's faster" into "sectors/req fell 31→4, DRAM rose 22%→81%".
Clock locking
Boost clocks vary run to run and with temperature; lock SM/memory clocks (nvidia-smi -lgc, ncu's clock control) for reproducible A/B numbers — or your "regression" may be thermal noise.

Core concepts

Timeline first, kernel second

Optimizing a kernel that is 8% of the wall time caps your win at 8% (Amdahl). nsys shows where time actually goes: kernels, memcpys, or gaps where the GPU waits on the CPU. Only once a kernel dominates do you point ncu at it. Re-profile after every fix — the bottleneck moves.

The SOL triage

SM% high → compute-bound: cheaper/lower-precision math, tensor cores, better algorithm. Memory% high → memory-bound: find the saturated level and reduce its traffic. Both low → latency-bound: more parallelism (occupancy, ILP, several elements per thread), fix divergence, split serialized atomics. Both high → the kernel is balanced and near the roof; only algorithmic change helps. This one chart picks which section of the playbook applies.

Reading L2 metrics honestly

High L2 hit rate is good news only if L2 traffic matters. Pure streaming kernels (each byte used once) show low hit rates — expected, not fixable, and not worth chasing. Conversely a great hit rate can hide a bug: massively redundant loads that L2 absorbs still burn L2 bandwidth — check whether L2 throughput itself is the saturated ceiling. The honest chain: which level is nearest peak → what traffic hits it → is that traffic necessary?

Occupancy is a means, not a score

Occupancy buys latency hiding. If DRAM is already at 90% of peak, more warps change nothing — the kernel is bandwidth-bound, not latency-bound. If both SOLs are low and Long Scoreboard stalls dominate, then raising occupancy (or ILP per thread) is exactly the fix. Also compare theoretical vs achieved: a gap means imbalance or tail effect — fix the launch shape, not the register count.

Stall reasons name the culprit

Long Scoreboard = warps parked on global-memory dependencies → coalesce, cache, prefetch, add ILP. Short Scoreboard = shared-memory pressure/bank conflicts. Barrier = threads waiting at __syncthreads() → unbalanced work or oversized blocks. LG/MIO Throttle = memory-op queues full → too many small requests; vectorize. "Not Selected" is healthy — it means there were spare eligible warps. Stall data turns "memory-bound" from a shrug into a specific edit.

Back-of-envelope beats guessing

A kernel moves 800 MB in 10 ms → 80 GB/s. On a 900 GB/s part that's 9% of peak: latency or access-pattern problem, not bandwidth. At 700 GB/s it's near the roof: only moving fewer bytes (fusion, reuse, smaller types) can help. Same for FLOPs vs peak. Do this arithmetic before and after every change — it catches both fake wins and fake losses.

4. The Optimization Playbook

The standard moves, each tied to the bottleneck it removes: tiling and fusion for DRAM traffic, ILP and vectorization for latency, shuffles and privatization for serialization, tensor cores and PTX for compute, libraries before hand-rolled everything.

Key terms

Tiling
Partition data into tiles staged in shared memory (or registers) so each global byte is loaded once and reused many times — raising arithmetic intensity.
Kernel fusion
Merging consecutive elementwise/short kernels into one so intermediates stay in registers instead of making a DRAM round trip per stage.
Grid-stride loop
for (i = blockIdx.x*blockDim.x + threadIdx.x; i < n; i += gridDim.x*blockDim.x) — decouples grid size from problem size, handles any n, and lets one thread process several elements (ILP for free).
ILP (instruction-level parallelism)
Independent operations per thread that the pipeline can overlap. Processing 4 elements per thread hides latency even at low occupancy — occupancy's complement.
Loop unrolling
#pragma unroll with compile-time trip counts: removes loop overhead, enables register allocation of array elements, exposes ILP. Needs bounds known at compile time — hence template tile sizes.
Warp shuffle
__shfl_down_sync/__shfl_xor_sync: exchange registers between lanes of a warp directly — no shared memory, no __syncthreads(). The fast lane for warp-level reductions and scans.
Privatization
Give each block (or warp/thread) a private copy of a contended structure — e.g. per-block histogram in shared memory — then merge once at the end, turning a global atomic hotspot into cheap local updates.
__launch_bounds__ / maxrregcount
Caps on registers per thread so more warps fit. The tradeoff: too tight a cap forces spills to local memory — verify with ncu that occupancy gained > spill traffic lost.
Tensor cores
Matrix-multiply-accumulate units operating on small tiles (FP16/BF16/TF32/FP8, FP4 on Blackwell; FP32 accumulate). Reached via cuBLAS/cuDNN/CUTLASS, the wmma API, or mma/tcgen05 PTX.
Inline PTX
asm volatile("…") embedding GPU assembly for instructions the compiler won't emit or ordering it won't preserve. Last resort; document why.
CCCL (CUDA C++ Core Libraries)
Thrust (host-callable device-wide algorithms), CUB (warp/block/device building blocks you compose inside kernels: BlockReduce, BlockScan, DeviceRadixSort), libcu++ (cuda::std:: atomics, barriers, pipelines).
Streams & CUDA Graphs
Streams order work and enable copy/compute overlap; graphs record a whole launch DAG and replay it with one call, killing per-launch CPU overhead for many small kernels.
Double buffering
Ping-pong two shared-memory tiles: while computing on tile A, asynchronously load tile B (cp.async/TMA), then swap — overlapping the load latency with math.
Thread coarsening / register blocking
Each thread computes a small tile of outputs held in registers (e.g. 8×8), amortizing every shared/global load over more math — the standard step between shared-memory tiling and tensor-core kernels.
Warp specialization
Different warps of one block take different roles — producer warps drive TMA/cp.async loads while consumer warps compute, coordinated with barriers. The Hopper/Blackwell GEMM pipeline pattern.
Persistent kernel
Launch exactly enough blocks to fill the GPU once and loop over a work queue inside the kernel — removes per-launch overhead and keeps caches warm for streams of small tasks.
cudaOccupancyMaxPotentialBlockSize
Runtime API that suggests the block size maximizing occupancy for a kernel on the current GPU — a sane starting default; still measure, because maximum occupancy ≠ maximum performance.

Core concepts

Match the move to the measurement

Every technique attacks one bottleneck: DRAM-bound → tiling, fusion, SoA, smaller types; latency-bound → ILP, occupancy, grid-stride with multiple elements; serialization → shuffles, privatization, fewer atomics on one address; compute-bound → tensor cores, lower precision, better algorithm; launch-bound → fusion and graphs. Applying a technique without the measurement is how kernels get "optimized" slower.

Tiled matmul, the canonical pattern

Naive SGEMM reads each input element N times from global memory. Tile it: each block loads a T×T tile of A and B into shared memory (coalesced), __syncthreads(), computes the partial product with each element reused T times, __syncthreads() again, next tile. AI rises from ~0.25 to ~T/4 FLOP/byte — from hopeless to compute-bound. Every further refinement (register blocking, double buffering, tensor cores) builds on this skeleton.

Reductions done right

Tree-reduce in shared memory down to 32 values with strides that keep active threads contiguous (avoids divergence), then finish the last five steps with __shfl_down_sync — no barriers needed inside a warp. Combine per-block partials with one atomicAdd per block or a second tiny kernel. CUB's BlockReduce packages exactly this; using it is usually the professional answer.

Libraries before heroics

cuBLAS/cuFFT/cuDNN, CUTLASS templates, and CUB primitives are tuned per architecture by people with the hardware manuals. The professional pattern: compose libraries for the heavy math, write custom kernels only for the glue between them — and fuse that glue. Hand-writing a GEMM that loses 3× to cuBLAS is a rite of passage, not a deliverable.

Compile-time knowledge is performance

Template parameters for tile sizes and flags turn runtime values into compile-time constants: static shared arrays get sized, loops fully unroll, per-thread arrays live in registers, dead branches vanish under if constexpr. The cost is one kernel instantiation per configuration — the standard trick behind CUTLASS's and CUB's performance.

Know when you're done

A memory-bound kernel at ~85% of DRAM peak with sectors/req ≈ 4 is finished — further tuning is noise; only an algorithmic change (move fewer bytes) can help. A compute-bound kernel near SM peak on the right pipes is likewise done. Write that down with the metrics as evidence: "done, and here's why" is a deliverable in profiler-guided work.

Tiled matrix multiply — one block's loop

__shared__ float As[T][T], Bs[T][T+1];            // +1: avoid bank conflicts
float acc = 0;
for (int k0 = 0; k0 < N; k0 += T) {
    As[ty][tx] = A[row*N + k0+tx];                // coalesced global load
    Bs[ty][tx] = B[(k0+ty)*N + col];
    __syncthreads();                              // tile fully loaded
    #pragma unroll
    for (int k = 0; k < T; ++k)
        acc += As[ty][k] * Bs[k][tx];             // each element reused T times
    __syncthreads();                              // done reading before overwrite
}
C[row*N + col] = acc;

each global element: loaded once per tile instead of once per use
arithmetic intensity: ~0.25 → ~T/4 FLOP/byte  (T=32 → 8: compute-bound)

Warp reduction with shuffles — last 5 steps, no barriers

v += __shfl_down_sync(0xffffffff, v, 16);   lanes 0-15  ← lanes 16-31
v += __shfl_down_sync(0xffffffff, v, 8);    lanes 0-7   ← lanes 8-15
v += __shfl_down_sync(0xffffffff, v, 4);
v += __shfl_down_sync(0xffffffff, v, 2);
v += __shfl_down_sync(0xffffffff, v, 1);    lane 0 holds the warp sum

register-to-register exchange: no shared memory, no __syncthreads()
block sum: warp sums → shared[32] → first warp reduces them the same way

Double buffering — hide the load behind the math

single buffer:   [load T0]→[compute T0]→[load T1]→[compute T1]→ ...
                  SM idle during every load

double buffer:   [load T0]→[compute T0 ‖ cp.async T1]→[compute T1 ‖ load T2]→
                  loads ride under compute; steady state hides them entirely
                  (Hopper/Blackwell: TMA moves whole tiles without using threads)

5. C++17 for Kernel Work

The role asks for fluency "through C++17". What matters in practice: the compile-time toolkit (templates, constexpr, if constexpr) that makes kernels fast, the vocabulary types (optional, variant, string_view), and the CUDA-specific rules for lambdas and constexpr on device.

Key terms

if constexpr
Compile-time branch: the false branch is discarded at instantiation, not merely optimized — code in it may even be invalid for the current template arguments (if dependent). Zero runtime cost.
constexpr functions
Evaluable at compile time when arguments are constant; also usable in device code. --expt-relaxed-constexpr lets device code call host constexpr functions (e.g. parts of the standard library).
Non-type template parameters
template<int TILE> __global__ void k(…): the tile size becomes a compile-time constant — static shared arrays, full unrolling, register allocation. One instantiation per configuration.
Structured bindings
auto [lo, hi] = minmax(…): unpack pairs/tuples/structs into named variables.
Fold expressions
(args + ...): apply an operator across a parameter pack without recursion — variadic sums, all-of checks, apply-to-each.
std::optional
Value-or-empty without sentinels or out-params. std::variant = type-safe union; std::string_view = non-owning view of a string, no allocation, cheap slicing.
CTAD
Class template argument deduction: std::pair p{1, 2.0} deduces pair<int,double>.
Inline variables
inline constexpr int N = 128; in a header — one entity across all TUs, no ODR violations for header-only config constants.
[[nodiscard]]
Warns when a return value is ignored — the natural annotation for functions returning cudaError_t-style status codes.
Extended / device lambdas
nvcc --extended-lambda allows __device__-annotated lambdas passed to kernels or Thrust. Capture by value: a by-reference capture of host stack memory is garbage on the device.
type_traits / static_assert
Fully usable in device code: constrain template kernels (static_assert(std::is_floating_point_v<T>)) and dispatch on type properties at compile time.
cuda::std (libcu++)
Heterogeneous standard library: cuda::std::atomic, cuda::std::barrier, tuples, type traits — the same types usable in host and device code.
cudaError_t discipline
Every API call returns a status, and kernel launches report errors asynchronously — check cudaGetLastError() after the launch and a sync in tests, or failures surface later at some unrelated call. Wrap it all in a CUDA_CHECK macro.
RAII device buffers
std::unique_ptr with a cudaFree deleter (or a small owning wrapper) makes device allocations leak-proof across early returns and exceptions — the host-side hygiene assessments look for.
constexpr lambdas
C++17 lambdas are implicitly constexpr where possible, so they can produce compile-time values used for tile geometry and static_assert checks.
std::visit
Applies a visitor over a std::variant's alternatives — the type-safe replacement for tag-switch unions in host-side config and dispatch code.

Core concepts

if constexpr replaces specialization boilerplate

Pre-17, variant behavior in a template kernel meant tag dispatch or partial specialization. Now one kernel body reads naturally: if constexpr (UseBias) { acc += bias[i]; } — the false case compiles to nothing, registers aren't spent on it, and both variants live in one function. Combined with non-type template parameters this is the backbone of modern kernel libraries.

Compile-time constants are a performance feature

__shared__ float tile[T][T+1] requires T at compile time; #pragma unroll fully unrolls only known trip counts; a per-thread array indexed by a loop variable stays in registers only if the loop unrolls. That's why kernels take sizes as template parameters and hosts dispatch over a small set of instantiations rather than passing sizes as runtime arguments.

Value semantics at the boundary

Kernel arguments are copied by value to the device (via constant memory), so pass small structs freely — but never pointers to host stack, and never lambdas capturing by reference. The same rule shapes device lambdas: capture by value, or capture device pointers explicitly. A std::vector can't cross the boundary at all — pass the .data() device pointer and size.

Host-side C++17 hygiene

The host code around kernels is ordinary modern C++: RAII wrappers for cudaMalloc/cudaFree (unique_ptr with a custom deleter is idiomatic), [[nodiscard]] status types with a CUDA_CHECK macro, string_view for parsing, optional for maybe-results, structured bindings for multi-returns. Assessments probe this: C++17 fluency means the host harness too, not just the kernel.

One template kernel, many tuned variants

template <int TILE, bool RELU>
__global__ void fused_gemm_act(const float* A, const float* B, float* C, int n) {
    __shared__ float As[TILE][TILE + 1];        // TILE must be compile-time
    ...
    #pragma unroll                              // fully unrolls: TILE is constant
    for (int k = 0; k < TILE; ++k) acc += As[ty][k] * Bs[k][tx];
    if constexpr (RELU) acc = fmaxf(acc, 0.f);  // false → not even compiled
    ...
}
// host dispatch over a closed set of instantiations:
if (tile == 32) fused_gemm_act<32, true><<<g, b>>>(A, B, C, n);

Call flow — a well-structured host harness (RAII + error checks)

run_gemm(h_A, h_B, n)
 ├─ DeviceBuffer d_A(n), d_B(n), d_C(n)     // RAII: cudaMalloc in ctor,
 │                                          //       cudaFree in dtor
 ├─ CUDA_CHECK( cudaMemcpyAsync(d_A, h_A, …, stream) )
 ├─ CUDA_CHECK( cudaMemcpyAsync(d_B, h_B, …, stream) )
 ├─ gemm<<<grid, block, 0, stream>>>(d_A, d_B, d_C, n)

6. Portability & Modern Hardware — HIP, Shaders, Blackwell

The concepts transfer: warp ≈ wavefront ≈ wave/subgroup, shared memory ≈ LDS ≈ groupshared. What changes are the widths, the tools, and the generation-specific features worth knowing by name.

Key terms

HIP
AMD's CUDA-shaped runtime and kernel language; compiles for AMD (ROCm) and NVIDIA. API maps ~1:1: cudaMalloc→hipMalloc, <<<…>>> launches work unchanged.
hipify
Source-to-source translators (hipify-perl, hipify-clang) that convert CUDA code to HIP; most kernels port with little or no manual editing.
Wavefront
AMD's warp: 64 lanes on CDNA (MI-series), 32 on RDNA. Portable warp-level code uses warpSize instead of a hardcoded 32.
LDS
Local Data Share — AMD's shared memory. Same role, same bank-conflict reasoning.
rocprof / Omniperf
AMD's profiling stack; Omniperf gives ncu-style per-kernel analysis (memory chart, roofline, wavefront occupancy).
HLSL compute shader
[numthreads(x,y,z)] = block size; groupshared = shared memory; SV_GroupThreadID/SV_DispatchThreadID = threadIdx / global id; GroupMemoryBarrierWithGroupSync() = __syncthreads.
Wave intrinsics (SM6+)
HLSL's warp-level ops: WaveActiveSum, WaveReadLaneAt, WaveGetLaneIndex — shuffles/ballots by another name. Vulkan/GLSL calls them subgroup operations.
Slang
Modular shading language (modules, generics, interfaces) that compiles to HLSL/DXIL, SPIR-V, Metal, and CUDA — one kernel source targeting many APIs.
Thread block clusters
Hopper+: groups of blocks co-scheduled on one GPC that can access each other's shared memory (distributed shared memory) and sync — a new tier between block and grid.
TMA (Tensor Memory Accelerator)
Hopper/Blackwell hardware engine for bulk asynchronous global↔shared tile copies — no threads or registers spent on address math; pairs with cuda::barrier/pipelines.
Blackwell (SM100/SM120)
Current NVIDIA generation (B200/GB200, RTX 50). Fifth-gen tensor cores add FP4/FP6 with microscaling formats; bigger L2; NVLink 5; data-center parts add dedicated tensor memory for MMA accumulators (tcgen05 PTX).
Compute capability
NVIDIA's architecture version (Ampere 8.x, Ada 8.9, Hopper 9.0, Blackwell 10.x/12.x). Compile fat binaries with PTX fallback so kernels run — then re-tune — on newer parts.
Fat binary & PTX JIT
nvcc -gencode embeds SASS for chosen architectures plus PTX; on a newer GPU the driver JIT-compiles the PTX. Code runs forward, but re-tune it — JIT'd PTX won't exploit new hardware features by magic.
warpSize
Built-in variable (CUDA and HIP) holding the actual lane width — the portable spelling of "32" for masks, shuffle offsets, and per-warp scratch sizing.
SPIR-V / DXIL
The binary shader IRs consumed by Vulkan and D3D12 respectively; GLSL/HLSL/Slang compile to them — the shader world's PTX analogue.
hipBLAS vs rocBLAS
rocBLAS is AMD's native BLAS; hipBLAS is the thin portability layer dispatching to rocBLAS or cuBLAS — the library half of a HIP port.

Core concepts

One mental model, many spellings

SIMT lanes grouped in warps, blocks with scratchpad and a barrier, a grid of independent blocks: CUDA, HIP, HLSL, GLSL, and Slang all expose exactly this. Learn the mapping table once and a tiled reduction ports across all five with renames. The genuinely portable skills are the invariants: coalescing, bank conflicts, occupancy, divergence — every vendor's profiler measures the same physics.

The width trap

The classic porting bug: warp-level code assuming 32 lanes (shuffle reductions, ballot masks, tile[32] arrays) silently breaks on CDNA's 64-wide wavefronts — half the data goes missing. Write warpSize-parameterized code, or template the width. Same trap in shaders: wave size varies by vendor (32/64) and Vulkan lets you query — never assume.

Blackwell in one paragraph

For an optimizer, Blackwell means: tensor cores now reach FP8 and FP4 (with per-block scaling factors — "microscaling") for inference throughput; TMA + clusters from Hopper remain the way to feed them; L2 keeps growing (tile accordingly); and on data-center parts, MMA accumulators move to dedicated tensor memory instead of registers. Practical takeaway: peak numbers assume tensor-core paths — a scalar-FP32 kernel leaves ~10× on the table on these parts.

Optimizing without owning the algorithm

The job description says it directly: improve kernels "without extensive background in the underlying algorithms." The method: treat the kernel as a memory-and-instruction stream — the profiler tells you traffic shapes, stall reasons, and pipe utilization without any domain knowledge. Preserve semantics with golden-output tests before touching anything; then every classified bottleneck has a fix that is algorithm-agnostic (layout, tiling, fusion, precision, launch shape).

7. Case Studies — Principal-Level Drills

Whole-situation questions, the way the work actually arrives: a full profiler report to diagnose, a kernel you didn't write to review, a teammate's claim to check, a regression to triage. Each case bundles several concepts — the skill under test is deciding which evidence matters and which is noise.

How to approach these

Evidence before edits

Every conclusion should trace to a specific number in the report — and the numbers that are absent or normal rule fixes out just as firmly. A diagnosis the metrics don't support is a guess wearing a suit; half of these cases contain a deliberately irrelevant metric.

Correctness is non-negotiable

You'll often optimize kernels whose algorithms you don't fully know. Golden outputs captured before the first change, re-validated after every change — including edge sizes (n not a multiple of the block). A 4× speedup that's 1% wrong is worth exactly nothing.

Know when you're done

At the roofline with clean access patterns, the honest deliverable is "finished, and here's the evidence" plus what an algorithmic change would buy. Saying no to further tuning — with metrics — is a principal-level answer, not a cop-out.
